## 1. 已确认的设计结论

1. 保留现有单篇问答接口，新增独立的全库问答接口，由 Controller 路径区分两种模式。
2. 不让前端通过“可选 postId”控制同一个接口，避免接口含义模糊。
3. `RagQueryService` 分别提供单篇回答入口和全库回答入口，再复用内部回答流程。
4. `RagRetrievalService` 分别提供单篇检索入口和全库检索入口，再复用内部检索流程。
5. 内部检索根据 `postId` 是否存在决定是否添加向量库过滤器：
   - 有 `postId`：只检索指定知文；
   - 无 `postId`：不添加知文 ID 过滤器，检索整个向量库。
6. 这不是方法重写。可以用方法重载实现，但本项目优先使用含义明确的方法名，减少误用。
7. `topK` 保留为前端可调参数，用于观察不同召回数量对 RAG 效果的影响，并继续限制在 `1-20`。
8. `maxTokens` 改为后端配置，不再由前端传入。
9. 本轮只扩展检索范围，继续复用已经完成的 HyDE 和 RRF，不加入 Rerank、查询改写或同义词扩展。

## 2. 目标接口

| 场景 | 请求路径 | 请求参数 | 说明 |
|---|---|---|---|
| 单篇问答 | `GET /api/v1/knowposts/{id}/qa/stream` | `question`、`topK` | 保留现有能力，按 `postId` 过滤 |
| 全库问答 | `GET /api/v1/knowposts/qa/stream` | `question`、`topK` | 新增能力，不按 `postId` 过滤 |

共同约束：

- `question` 必填，去除首尾空格，最大长度保持 500；
- `topK` 默认值保持 5，允许范围保持 `1-20`；
- 返回类型继续使用当前的 `Flux<String>` SSE 格式，避免本轮同时修改前端流式解析协议；
- `maxTokens` 不再出现在请求参数中。

两个接口的路径由 Spring MVC 直接区分，不需要 Controller 再判断前端是否传了 ID。

## 3. 目标调用链

### 3.1 单篇问答

`KnowPostRagController` 单篇入口
→ `RagQueryService` 单篇回答入口
→ `RagRetrievalService` 单篇检索入口
→ 确认该知文已建立索引
→ 原问题检索与 HyDE 检索都添加 `postId` 过滤器
→ RRF 融合
→ GPT 根据融合结果生成答案。

### 3.2 全库问答

`KnowPostRagController` 全库入口
→ `RagQueryService` 全库回答入口
→ `RagRetrievalService` 全库检索入口
→ 不执行单篇 `ensureIndexed(postId)`
→ 原问题检索与 HyDE 检索都不添加 `postId` 过滤器
→ RRF 融合
→ GPT 根据融合结果生成答案。

两条链路只有“索引准备方式”和“检索过滤范围”不同，HyDE、RRF、上下文拼接和 GPT 流式生成继续共用。

## 4. 后端文件改造清单

### 4.1 `KnowPostRagController`

修改内容：

- 保留现有单篇 SSE 接口；
- 新增全库 SSE 接口；
- 单篇和全库接口都只接收 `question` 与 `topK`；
- 从现有单篇接口移除 `maxTokens` 请求参数；
- 分别调用 `RagQueryService` 的单篇入口和全库入口。

职责边界：Controller 只负责路由、参数校验和选择业务入口，不构造向量过滤器。

### 4.2 `RagQueryService`

修改内容：

- 将现有单篇回答入口改为含义明确的单篇方法；
- 新增全库回答入口；
- 两个公开入口复用内部回答流程；
- 单篇入口调用单篇检索，全库入口调用全库检索；
- 根据查询范围调整空召回提示：单篇提示“当前文章”，全库提示“知识库”；
- 从后端配置读取 `maxTokens`，并真正应用到模型生成选项。

职责边界：只负责取得检索结果、构造上下文和提示词、调用 GPT，不负责向量过滤规则。

现状注意：当前 `streamAnswerFlux` 虽然接收 `maxTokens`，但并未将它设置到 `OpenAiChatOptions`，因此该参数目前实际没有生效。本轮需要一起纠正。

### 4.3 `RagRetrievalService`

修改内容：

- 将现有 `retrieve` 明确为单篇检索入口；
- 新增全库检索入口；
- 两个公开入口复用内部检索流程；
- 内部流程使用可选的 `Long postId` 表示检索范围；
- 只有单篇检索执行 `indexService.ensureIndexed(postId)`；
- `searchDocuments` 根据 `postId` 是否存在，选择添加过滤器或构建无过滤器的 `SearchRequest`；
- 原问题和 HyDE 假设答案必须使用相同的检索范围；
- 日志增加明确的 `scope=post/global`，全库模式不打印伪造的 postId。

职责边界：负责原问题检索、HyDE 检索和 RRF 融合，不负责最终答案生成。

### 4.4 RAG 配置

新增或修改内容：

- 增加后端 RAG 回答配置，例如 `rag.answer.max-tokens`；
- 默认值建议先保持 1024；
- 通过项目统一的配置属性类注入 `RagQueryService`；
- 配置值需要设置合理下限和上限，避免错误配置导致响应异常。

本轮 `topK` 暂不收回配置文件，继续由前端传入并由 Controller 限制范围。

### 4.5 `SecurityConfig`

修改内容：

- 现有匿名规则只覆盖单篇路径；
- 如果首页全库问答允许匿名使用，需要增加全库 SSE 路径的 GET 放行规则；
- 调试接口和手动重建接口不随之公开。

安全提醒：公开的全库 LLM 接口后续应增加限流。本轮可以先保持与现有单篇接口一致，但上线前必须补齐。

## 5. 全库索引前置条件

单篇问答可以在请求开始时执行 `ensureIndexed(postId)`，因为它知道具体知文 ID。全库问答没有具体 ID，不能在每次提问时遍历数据库并重建全部索引。

当前项目已经在以下生命周期中维护向量索引：

- 知文发布后执行预索引；
- 知文更新为公开时执行预索引；
- 知文变为非公开时删除向量；
- 知文删除时删除向量。

因此，全库检索应直接使用已经维护好的向量库。为了避免历史知文漏建索引，正式测试前需要执行一次全量补建或确认现有公开已发布知文均已入库。这个补建动作属于运维或索引管理能力，不应放进在线问答请求链路。

## 6. 调试与观测改造

本轮主功能只要求全库回答接口，但为了继续观察 HyDE 和 RRF 效果，建议同步扩展调试能力：

- 保留现有单篇调试入口；
- 新增全库调试入口；
- `RagDebugService` 与 `RagQueryService` 一样调用公共的 `RagRetrievalService`；
- 调试 DTO 将 `postId` 改为可空，或增加明确的 `scope` 字段；
- 每个召回切片继续返回 `chunkId`、标题、位置、向量分数和内容预览；
- 全库结果必须能看到每个切片所属的 `postId`，否则无法判断跨文章召回是否正确。

调试能力不参与 GPT 回答，只展示原问题召回、HyDE 召回和 RRF 融合的中间结果。

## 7. 测试改造清单

### 7.1 `RagRetrievalServiceTest`

需要验证：

- 单篇检索会执行 `ensureIndexed(postId)`；
- 单篇原问题检索和 HyDE 检索都带同一个 `postId` 过滤器；
- 全库检索不会执行单篇索引确认；
- 全库原问题检索和 HyDE 检索都不带 `postId` 过滤器；
- HyDE 为空时只使用原问题结果；
- 两路都有结果时继续执行 RRF，并只保留 `topK` 个结果。

### 7.2 `RagQueryServiceTest`

需要验证：

- 单篇回答调用单篇检索入口；
- 全库回答调用全库检索入口；
- 两种模式都使用融合后的文档构造上下文；
- 单篇空召回和全库空召回返回不同且准确的提示；
- 后端 `maxTokens` 配置被应用，而不是仅被读取。

### 7.3 Controller 与安全测试

需要验证：

- 原单篇 URL 继续可用；
- 新全库 URL 能返回 SSE；
- `question` 与 `topK` 校验保持有效；
- 请求中的 `maxTokens` 不再作为业务参数；
- 全库接口的匿名或登录策略与产品决定一致。

### 7.4 真实效果测试

准备至少三篇主题有交叉、内容有差异的公开知文，提出以下问题：

- 答案只存在于一篇文章中的问题；
- 答案需要综合两篇文章的问题；
- 与知识库无关的问题；
- 表述模糊但 HyDE 可能改善召回的问题；
- 调整不同 `topK` 后容易引入噪声的问题。

观察原问题召回、HyDE 召回、RRF 顺序和最终答案是否一致，不能只观察 GPT 最终回答。

## 8. 建议实施顺序

1. 增加后端 `maxTokens` 配置，并明确 Query 层的回答参数。
2. 改造 `RagRetrievalService`，先打通单篇与全库两种检索范围。
3. 改造 `RagQueryService`，增加两个清晰的回答入口并复用生成流程。
4. 在 `KnowPostRagController` 增加全库 SSE 入口，同时调整单篇入口参数。
5. 根据产品访问策略更新 `SecurityConfig`。
6. 扩展调试 DTO 和全库调试入口，验证 HyDE 与 RRF 的中间结果。
7. 补齐单元测试，再连接真实向量库和模型服务做效果测试。
8. 后端验证稳定后，再在前端首页增加全库问答入口。

---

## 9. 本轮不做

- 不加入 Rerank 模型；
- 不加入查询改写和同义词扩展；
- 不改变 HyDE 提示词；
- 不改变 RRF 算法；
- 不升级 SSE 消息协议；
- 不在在线请求中执行全库重建；
- 不进行大规模包结构调整；
- 不直接实现首页前端界面。

这些能力应在全库检索的可观测测试完成后，根据真实召回问题再决定优先级。
